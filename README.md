## 1. 痛点

在 **汽车** 行业——尤其是 **智能驾驶、车载知识问答**（这里只是举例，同类需求也出现在试验规范、标定、诊断等文档里）——大家希望借助大模型回答：

- 这个工况速度阈值是多少？  
- 某种误触发怎么判定？  
- 两条试验要求差在哪里？  

但这类问题的答案，往往对应 **已经写死在文档里的确定知识**，不是开放聊天。

```text
  用户提问（阈值 / 工况 / 判据）
           │
           ▼
     ┌───────────┐     若靠「模型背」
     │  大模型    │ ──────────────────► 可能编造条款、改数字
     └───────────┘                      听起来对，工程上危险
```

所以核心要求是：

**必须严格按照已有的、可核对的确定知识来回答；材料不够就明确说不知道，绝不能硬编。**

## 2. 从传统 RAG，到加上 Agent

### 2.1 为什么会想到 RAG

为了让模型「有据可依」，常见做法是 **RAG（检索增强生成）**：先从知识库里找出相关文档片段，再让模型基于这些片段生成回答。

```text
        问题
          │
          ▼
     ┌─────────┐
     │  检索   │  从文档库找相关片段
     └────┬────┘
          ▼
     ┌─────────┐
     │  生成   │  基于片段写答案
     └─────────┘
```

这比纯靠模型记忆好，但传统用法通常是 **单次直通**。

### 2.2 传统 RAG：单次「问题 → 检索 → 回答」

```text
┌──────┐    ┌──────────────┐    ┌──────────┐
│ 问题 │───►│ 检索一次     │───►│ 直接回答 │
└──────┘    │ 片段定生死   │    │ 容易硬编 │
            └──────────────┘    └──────────┘
```

在汽车长文档里，一轮往往不够：只搜一次容易漏；表格被切碎会答错阈值；噪声会诱发脑补；证据不足仍硬答；也难说清出处。

### 2.3 本项目：加上 Agent 控制环

相对传统 RAG，关键不是「多接一个库」，而是中间加了 **Harness Agent**：

```text
传统 RAG（单次）:

  问题 ──────► 检索 ──────► 回答


EvidenceTrail（多步）:

              ┌──────────────────────────────┐
              │         Harness Agent        │
  问题 ──────►│  规划 → 检索 → 反思 → 再决策  │──────► 有据回答
              │         ↑____________│       │         或拒答
              │         不够就再搜            │
              └──────────────┬───────────────┘
                             │ 调用
                             ▼
                    ┌─────────────────┐
                    │ 图 + 向量检索底座 │
                    └─────────────────┘
```

| | 传统 RAG | EvidenceTrail |
|--|----------|----------------|
| 流程 | 单次直通 | 多轮：规划 → 检索 → 反思 |
| 谁决定搜什么 | 基本固定整句问题 | Agent 写子问题、选方式 |
| 证据不够 | 常硬答 | 继续取证，或明确拒答 |
| 工程落点 | 检索 + 一次生成 | 检索底座 + Harness |

> 传统 RAG = 一次检索生成；本项目 = 在 RAG 上加了 Agent，把问答做成可规划的取证过程。

### 2.4 知识图谱、GraphRAG 和 LightRAG

**知识图谱**：概念是节点，联系是边。比纯向量更适合串起「要求—试验—条件—阈值」这类结构关系。

**GraphRAG**：检索时不只做向量相似，还用图做关联扩展，再回到原文给模型生成。

```text
  文档 ──► 切分 / 抽取 ──► 知识图谱 + 向量索引
                                │
                     查询时：图上扩展关联
                                │
                                ▼
                           回到原文片段
                                │
                                ▼
                              生成答案
```

本项目底座用开源 **[LightRAG](https://github.com/HKUDS/LightRAG)**：切分、抽实体关系、写图库和向量库、提供图增强/纯向量等查询。  
它是通用底座；**领域里抽什么、关系允不允许、表格怎么切**，要在应用侧自己约束。本仓库不改 LightRAG 核心源码。

### 2.5 构图上做了哪些改进

通用 schema 偏「人物 / 组织 / 地点 / 概念」，关系也较自由，直接用在法规文档上容易：类型混、脏边多、数值没条件、表格被切碎。

本项目在样例域上的改进可以看成一条构图流水线：

```text
  原始文档
      │
      ▼
  ┌──────────────┐
  │ 结构切分      │  表格尽量整表 / 结构单元入库
  └──────┬───────┘
         ▼
  ┌──────────────┐
  │ 领域 schema  │  条款、试验、阈值、误响应等专用类型
  │ 抽取提示      │  见 config/gb_39901_2025_schema.yml
  └──────┬───────┘
         ▼
  ┌──────────────┐
  │ LightRAG 构图 │  写 Neo4j + 向量
  └──────┬───────┘
         ▼
  ┌──────────────┐
  │ 关系约束校验  │  限制允许的关系类型与端点
  └──────┬───────┘
         ▼
     可用知识库
         │
         │  查询时
         ▼
  图上广搜找路 ──► 回源完整原文 ──► 原文优先作答
```

相对通用做法：

| 点 | 改进 |
|----|------|
| Schema | 专用类型与约束，而不是宽泛通用标签 |
| 切分 | 表结构优先，减少表头与数据行分离 |
| 关系 | 提示 + 校验 + 后处理，减少类型不合法的边 |
| 检索用法 | 图找路、文答题；命中后补全文，不只给三元组 |

图更干净有助于检索，但 **不等于问答自动更好**，所以还要 Agent 取证和门控。  
样例是 GB 39901 AEBS；换文档域时主要重写 schema 和切分策略。

### 2.6 设计立场

```text
控制层：只定角色、工具、拒答、步数
        不定某题答案关键词
              │
              ▼
        Agent 在线取证作答 ──只读──► 索引 + 原文
              │
              │ 作答结束后
              ▼
        标准答案 / 参考证据 ──► 离线打分
        提问过程中不读
```

不会把某道题关键词写进规则刷分。题面知识在索引里；Agent 自己决定搜什么。

## 3. 用 Benchmark 做验证

能跑通 demo 不够，还要验证：检索有没有漏、图建得怎么样、回答是否忠于材料。  
因此单独做了离线评测，目录在 `benchmark/`。

### 3.1 验证框架

```text
                 ┌─────────────────┐
                 │ questions.jsonl │
                 └────────┬────────┘
                          │
        ┌─────────────────┼─────────────────┐
        ▼                 ▼                 ▼
 ┌────────────┐    ┌────────────┐    ┌────────────┐
 │ 图谱质量    │    │ 检索质量    │    │ 问答质量    │
 │ score_kg   │    │ score_     │    │ score_     │
 │            │    │ retrieval  │    │ answers    │
 └─────┬──────┘    └─────┬──────┘    └─────┬──────┘
       └─────────────────┼─────────────────┘
                         ▼
              ┌────────────────────┐
              │ 报告 / badcase     │
              │ 再改构图或 Agent    │
              └────────────────────┘
```

在线答题和离线打分必须分开：

```text
正确：先答后评

  问题 ──► Agent 只查索引/原文 ──► 回答 ──► 对照标准答案打分


错误：开卷答题（本项目不用）

  问题 ──► 把标准答案塞进提示或检索 ──► 再生成
```

### 3.3 例题

均来自 [questions.jsonl](benchmark/data/questions.jsonl)：

| 题型 | 题号 | 题目 |
|------|------|------|
| 直接事实 | `gb_direct_001` | GB 39901—2025 适用于哪两类汽车？ |
| 条件表格 | `gb_table_001` | M1 类试验车辆以 60 km/h、静止车辆目标、最大设计总质量状态试验时，允许的最大相对碰撞速度是多少？ |
| 多跳关系 | `gb_multi_hop_001` | 车辆目标碰撞预警能力应通过哪些试验验证，预警相对紧急制动的最迟时机是什么，并有什么例外？ |
| 比较例外 | `gb_compare_001` | M1 与 N1 对前方车辆目标的最低激活速度范围有何不同？对行人、自行车和踏板式两轮摩托车目标是否相同？ |
| 跨章节综合 | `gb_synthesis_001` | 完整列出 6.11 规定的五类误响应场景，并说明所有场景共同的合格判据。 |
| 故意不可答 | `gb_unanswerable_001` | 制造商能否完全用仿真替代 6.11 的五项误响应试验？如果可以，需要满足哪些条件？ |

最后一类检查系统会不会在材料推不出来时硬编。

### 3.4 标准答案怎么用

标准答案只用于跑完之后打分。在线 Agent 默认不加载评测数据。

### 3.5 跑下来看到的现象

```text
图检索 ──► 证据覆盖升高
     │
     └──► 噪声也可能升高 ──► 只靠多召回不够
                              │
                              ▼
                重排序 · 原文优先 · 门控 · 多步 Agent

结构切分 / 关系约束 ──► 图更干净、表题更稳
     │
     └──► 仍不等于问答自动更好 ──► 还要靠取证 Agent
```

完整 60 题冻结和统一总正确率还要再审题、重跑。当前交付的是可复现评测链路和 pilot/消融结论。  
细节见报告与 [PROJECT_STATUS.md](PROJECT_STATUS.md)。

## 4. 项目介绍：EvidenceTrail

一次提问的主流程：

```text
                    ┌─────────────────────┐
         ┌─────────►│ Agent 规划下一步     │◄─────────┐
         │          └──────────┬──────────┘          │
         │                     ▼                     │
         │          ┌─────────────────────┐          │
         │          │ 图/向量检索          │          │
         │          │ 命中后回源完整原文    │          │
         │          └──────────┬──────────┘          │
         │                     ▼                     │
         │          ┌─────────────────────┐          │
         │          │ 证据袋 + 重排序      │          │
         │          └──────────┬──────────┘          │
         │                     ▼                     │
         │              够不够？                      │
         │              /      \                     │
         │          不够        够了                  │
         │            │          │                   │
         └────────────┘          ▼                   │
                      ┌─────────────────────┐        │
                      │ 基于证据袋作答       │        │
                      │ 数字/空袋门控        │        │
                      └──────────┬──────────┘        │
                                 ▼                   │
                            输出 + 轨迹               │
```

目标：先取证，再作答；不够就拒答。

### 4.1 走题示意：6.11 五类误响应

```text
题目：列出 6.11 五类误响应 + 共同判据
        │
        ▼
  规划：图+向量搜 6.11 / 判据
        │
        ▼
  图命中 ──► 回源完整原文进袋
        │
        ▼
  袋里已有场景与判据？──否──► 换子问题再搜
        │
       是
        ▼
  基于证据作答 ──► 门控检查 ──► 输出
```

过程可用 `--dump-trace` 导出。

## 5. 我做了哪些内容

### 5.1 分层对应关系

```text
┌────────────────────────────────────────┐
│  harness/          Agent 控制环         │
├────────────────────────────────────────┤
│  lightrag_custom/  抽取提示、关系校验    │
│  config/           领域 schema          │
│  scripts/          切分、入库、后处理    │
├────────────────────────────────────────┤
│  LightRAG Docker   图 + 向量检索底座     │
│  Neo4j             图存储（本机）        │
├────────────────────────────────────────┤
│  benchmark/        离线验证              │
└────────────────────────────────────────┘
```

### 5.2 Harness 取证框架（`harness/`）

- 多步循环：每步选工具、写子查询  
- 只定规范、不定题解  
- 证据袋，原文优先  
- Top-K 和长度限制  
- 检索侧与袋侧重排序  
- 图命中后补全文  
- 判断是否够用，减少空转  
- 门控与轨迹导出  

### 5.3 构图与接入（`scripts/`、`lightrag_custom/`、`config/`）

- 领域 schema、结构切分、关系约束  
- Docker 使用 LightRAG，不改核心源码  

### 5.4 Benchmark（`benchmark/`）

- 题目与参考数据、分层打分脚本、报告  
- 只事后打分，默认不进在线提问  

### 5.5 目录

```text
harness/           # Agent
lightrag_custom/   # 接入定制
scripts/           # 预处理、入库
config/            # 领域 schema
benchmark/         # 评测
corpus/            # 样例语料
data/rag_storage/  # 样例最终版向量/KV 快照（约 31MB）
compose.yaml
Makefile
```

开源应用层；LightRAG 用镜像。不提交 `.env` 与 Neo4j 数据目录。

## 6. 环境配置与运行

### 6.1 部署关系

```text
你的机器
  ├── Docker
  │     ├── Neo4j          :7474 / :7687
  │     └── LightRAG       :9621   ◄── 挂载 data/rag_storage、lightrag_custom
  │
  └── Python
        └── reg_harness CLI / 代码  ──HTTP──► LightRAG
```

### 6.2 配置

需要：Docker、Python 3.10+、对话与向量 API。

```bash
git clone <本仓库>    # 建议名：evidence-trail
cd <本仓库目录>

pip install -r requirements.txt
cd harness && pip install -e . && cd ..

cp .env.example .env
# 填写 NEO4J_PASSWORD、LLM_*、EMBEDDING_*
# 可选 RERANK_*
```

样例可用 `.env.gb39901_v4` 作无密钥叠加配置。勿提交真实密钥。

### 6.3 运行步骤

```text
① 单元测试（可不启服务）
      cd harness && python3 -m unittest discover -s tests -v

② 若换自己的 PDF：先用 MinerU 转成 Markdown
      在线 https://mineru.net/
      或本地 https://github.com/opendatalab/MinerU
      结果放入 corpus/prepared/

③ 起服务
      make v4-up
      WebUI http://127.0.0.1:9621

④ 图为空则入库
      make v4-prepare && make v4-ingest …

⑤ 提问
      python3 -m reg_harness.cli --profile-env .env.gb39901_v4 ask "…"

⑥ 需要时离线打分
      benchmark/scripts/ …
```

提问示例：

```bash
cd harness

python3 -m reg_harness.cli --profile-env .env.gb39901_v4 \
  ask "GB 39901—2025 适用于哪两类汽车？" --max-steps 6

python3 -m reg_harness.cli --profile-env .env.gb39901_v4 \
  ask "完整列出6.11规定的五类误响应场景，并说明所有场景共同的合格判据。" \
  --max-steps 10

python3 -m reg_harness.cli --profile-env .env.gb39901_v4 \
  ask "你的问题" --dump-trace /tmp/evidence-trail.json
```

代码调用：

```python
from reg_harness import build_stack

stack = build_stack(profile_env=".env.gb39901_v4")
state = stack.ask("GB 39901 适用于哪两类汽车？", max_steps=6)
print(state.final_answer)
```

使用仓内向量快照时，向量模型与维度须与入库时一致。

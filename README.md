# EvidenceTrail

**基于知识图谱增强检索的文档取证 Agent**

---

## 1. 痛点

在 **汽车** 行业——尤其是 **智能驾驶、车载知识问答**（这里只是举例，同类需求也出现在试验规范、标定、诊断等文档里）——大家希望借助大模型回答：

- 这个工况速度阈值是多少？  
- 某种误触发怎么判定？  
- 两条试验要求差在哪里？  

但这类问题的答案，往往对应 **已经写死在文档里的确定知识**（表格数字、条款条件、合格判据），而不是开放聊天。

此时如果大模型产生 **幻觉**——捏造条款、改数字、混淆适用条件——句子仍然可能很通顺，后果却可能很严重：试验理解偏差、设计讨论跑偏，甚至触及安全相关误判。

所以核心要求是：

**必须严格按照已有的、可核对的确定知识来回答；材料不够就明确说不知道，绝不能硬编。**

## 2. 从传统 RAG，到加上 Agent

### 2.1 为什么会想到 RAG

为了让模型「有据可依」，常见做法是 **RAG（检索增强生成）**：先从知识库里找出相关文档片段，再让模型基于这些片段生成回答。这比纯靠模型背参数好。

### 2.2 传统 RAG：单次「问题 → 检索 → 回答」

传统 RAG 更像一条 **直通管道**，通常只跑 **一轮**：

```text
问题  ──►  检索（搜一批相关片段）  ──►  生成答案
              （一次定生死）              （直接出）
```

优点是简单；在汽车长文档里，一轮往往不够：

| 单次流程的短板 | 结果 |
|----------------|------|
| 只搜一次 | 跨条款、跨表的问题容易漏半截 |
| 切分不当 | 表头和数值行拆开，阈值答错 |
| 召回噪声 | 模型用「差不多相关」的段落脑补 |
| 没有「够不够」 | 证据不足也生成完整答案 → 幻觉出站 |
| 难追溯 | 用户不知道对应原文哪里 |

要「必须忠于确定知识」，既需要更好的检索组织，也需要中间有一层 **能规划、能补搜、能决定何时停手** 的控制——这就是 **Agent**。

### 2.3 本项目的关键增量：Agent（Harness 框架）

EvidenceTrail 相对传统 RAG 的首要优点，不是「多接了一个库」，而是把 **Agent 框架（代码里的 harness）** 嵌进链路：

```text
传统 RAG（单次）:
  问题 → 检索一次 → 回答

EvidenceTrail（多步 Agent）:
  问题
    →【Agent】想：还要不要搜？搜什么子问题？用图还是向量？
    → 检索 → 放进证据袋
    →【Agent】反思：够不够？不够就换问法再搜
    → 够了再基于证据袋回答
    → 代码检查：没证据 / 数字对不上原文 → 拒答
```

| | 传统 RAG | EvidenceTrail |
|--|----------|----------------|
| 流程 | **单次** 问题→检索→回答 | **多轮循环** 规划→检索→反思→再决策 |
| 谁决定搜什么 | 基本固定成整句问题 | **Agent 写子问题、选检索方式** |
| 证据不够时 | 仍常硬答 | **继续取证，或明确拒答** |
| 工程落点 | 检索 + 一次生成 | 检索底座 + **Harness 取证框架** |

> **传统 RAG = 一次检索生成；本项目 = 在 RAG 之上加了 Agent（Harness），把问答做成可规划的取证过程。**

### 2.4 检索思想：图谱广搜，再落到原文

在 Agent 循环内部，检索侧的想法是：

1. **在知识图谱上做广度搜索**——用实体、关系扩大「该去哪找」；  
2. **再定位到对应的原文片段**——用原文支撑事实和数字。  

**图负责找路，文负责答题。**  
重要细节：图上命中实体/关系之后，不能只把「三元组摘要」扔给模型——要 **回源到完整原文段落**（一个关系可能对应多段原文，都要尽量拉齐）。作答时也以 **原文段落为主**，图上的实体/关系只做导航，避免「图摘要听起来对、原文对不上」。

默认检索姿态倾向 **图+向量一起用**（完整信息都要），再用 **重排序** 压噪声。

> **实现说明：** 图谱构建与图+向量检索底座使用开源 [LightRAG](https://github.com/HKUDS/LightRAG)（Docker）。  
> 本仓库重点是其上的 **Harness 取证 Agent、入库定制与评测**，**不 fork 修改 LightRAG 核心源码**。

验证样例：**GB 39901-2025 轻型汽车 AEBS 法规**。方法 **不绑定** 该标准，可换其它技术文档。

### 2.5 设计立场

控制层只约定：Agent 要做取证、可以拒答、有步数上限、禁止无证据编造。  
**不会**把某道题的标准答案关键词写进规则里去刷分——那样等于瞄着靶子射箭，换一批题就失效。

题目相关的知识应在索引和原文里；Agent 根据领域说明和当前证据袋自己决定搜什么。

## 3. 用 Benchmark 做验证

能跑通 demo 不够，还要能回答：检索有没有漏、图建得怎么样、回答是否忠于材料。  
所以项目里单独做了 **离线评测集和打分脚本**，目录在 `benchmark/`。

### 3.1 评测覆盖什么

- **图谱**：实体关系是否合理、类型是否合法等  
- **检索**：该找回的证据找没找到、噪声多不多  
- **问答**：事实题、表格条件题、多跳和综合题，以及故意设计成「不该硬答」的题  

题目与参考数据在 `benchmark/data/`，题型包括直接事实、条件表格、跨条款、比较例外、跨章节综合、不可回答等。

### 3.2 标准答案怎么用

标准答案和参考证据只用于 **跑完之后打分、分析错在哪**。  
**不会在提问过程中**把它们塞进检索或提示词——否则等于开卷考试，分数不能代表用户随便问时的真实表现。

在线 Agent 默认不加载这套评测数据；只有 `benchmark/scripts` 一类离线打分会用到。

### 3.3 跑下来看到的现象

在 GB 39901 样例上做过入库对比和 pilot 问答，报告在 `benchmark/results/`。大致结论：

- 图检索往往能 **提高证据覆盖**，说明广搜有用。  
- 覆盖上去以后 **噪声也更大**，答案不一定同步变好，所以需要重排序、原文优先和门控，不能只靠多召回。  
- 加强关系约束后，类型不合法的关系可以明显下降；本地对比里曾从很多降到接近没有。  
- **图更干净，不等于问答自动更好**，多步取证和忠于原文仍然必要。  
- 表格按整表/结构切分后，表上的事实题可以稳定很多。  

完整 60 题冻结榜和可对外统一宣传的总正确率，还依赖审题和统一重跑。当前仓库交付的是 **可复现的评测链路** 和 pilot/消融结论，方便自己改完再验，而不是一份刷榜成绩单。  
更细的数字见 [pilot_6q_report.md](benchmark/results/pilot_6q_report.md)、[benchmark_report.md](benchmark/results/benchmark_report.md) 和 [PROJECT_STATUS.md](PROJECT_STATUS.md)。

## 4. 项目介绍：EvidenceTrail

**EvidenceTrail** = **Harness 取证 Agent** + 图增强检索 + 忠于原文的门控 + **离线 Benchmark 验证**。

一次完整提问大致是：

```text
用户问题
  →【Agent】规划：下一步用图还是向量、子问题怎么写（默认倾向图+向量一起）
  →【检索】召回；图命中则回源完整原文，进入证据袋
  →【Rerank】更相关的片段排前面（检索侧 + 袋内侧都可做，用来降噪）
  →【Agent 反思】袋里是否已经够？  
        · 不够 → 换问法再搜  
        · 其实够了却空转 → 代码侧也会收网，避免「觉得缺其实够」
  →【仅基于证据袋】生成结构化答案（原文段落优先，图摘要次之）
  →【门控】空袋拒答；关键数字必须在原文中出现
  → 输出结果 + 过程轨迹（可审计的取证过程）
```

目标始终是：**先取证，再作答；不够就拒答——而不是单次「搜一下就答」。**

### 4.1 走题示意（样例：6.11 五类误响应）

题目大意：列出 6.11 规定的五类误响应场景，并说明共同合格判据。

1. Agent 决定用 **图+向量** 搜「6.11 误响应 / 合格判据」类子问题；  
2. 图命中相关条款结构后，**回源拉完整原文**（而不只是关系短句）；  
3. 袋里已有场景列表与判据原文时，**应停止空转**，基于证据作答；  
4. 若数字/枚举与袋中原文不一致，门控会挡掉「漂亮但无据」的答案。  

（真实步数随模型与索引状态变化；过程可用 `--dump-trace` 导出。）

## 5. 我做了哪些内容

### 5.1 Harness 取证框架（`harness/`）——相对传统 RAG 的核心增量

| 内容 | 说明 |
|------|------|
| **多步 Agent 循环** | 相对单次管道：每步选工具、写子查询，可多轮补证 |
| **协议式控制** | 告诉模型「这层要做什么」（取证、可拒答、勿编造），而不是写死某题关键词剧本 |
| 证据袋 | 多轮累积；**原文段落优先** 进入作答上下文 |
| Top-K 与长度预算 | 限制召回与袋容量，减轻噪声 |
| **Rerank（检索 + 袋侧）** | 用排序模型压噪声，减少手写领域词加权 |
| **图 → 完整原文** | 命中实体/关系后回源 chunk（含多源拆分），作答给足上下文 |
| **反思是否够用** | 审袋：够了就作答；防「袋里已有却反复空搜」 |
| 忠于原文门控 | 空袋拒答；关键数字须在证据中出现 |
| 轨迹 | 每步可导出，支撑调试与 benchmark 对照 |

### 5.2 文档入库与图谱（`scripts/`、`lightrag_custom/`）

| 内容 | 说明 |
|------|------|
| 切分策略 | 表格尽量整表入库，减少切碎 |
| 关系约束 | 限制允许的关系类型，减少脏边 |
| 接入 LightRAG | 容器 + 定制完成建图、索引（**不改** LightRAG 核心源码） |

### 5.3 离线 Benchmark（`benchmark/`）——为验证而建

| 内容 | 说明 |
|------|------|
| 题目与参考材料 | `benchmark/data/`（金标 JSONL） |
| 打分脚本 | 图谱 / 检索 / 问答分层评分 |
| 报告 | pilot 与实验报告（见 `benchmark/results/*report*.md`） |
| 原则 | **事后验收**；默认不注入在线 Agent |

### 5.4 代码与目录

```text
harness/           # Agent 主逻辑（Harness）
lightrag_custom/   # 接入 LightRAG 的定制
scripts/           # 预处理、入库
benchmark/         # 评测集与打分（验证用）
corpus/            # 样例语料（prepared）
data/rag_storage/  # 样例最终版向量/KV 快照（约 31MB，不含图库）
compose.yaml       # Neo4j + LightRAG
Makefile           # 常用命令
```

- **贡献边界：** 开源的是应用层（Agent + 定制 + 评测）；LightRAG 以 Docker 依赖形式使用。  
- **不入库：** 含密钥的 `.env`、Neo4j 数据目录（约 500MB，本机重建）。  
- 示意图：[docs/architecture.svg](docs/architecture.svg)

## 6. 环境配置

**需要：** Docker、Python 3.10+、兼容 OpenAI 协议的对话模型与向量模型 API。

```bash
git clone <本仓库>    # 建议名：evidence-trail
cd <本仓库目录>

pip install -r requirements.txt
cd harness && pip install -e . && cd ..

cp .env.example .env
# 编辑 .env：NEO4J_PASSWORD、LLM_*、EMBEDDING_*
# 切勿提交真实密钥
```

可选：配置 `RERANK_*`，检索与证据排序会更稳。  
样例工作区可用 `.env.gb39901_v4` 作为无密钥配置叠加。

## 7. 如何运行

### 7.1 离线测试（可不启大模型服务）

```bash
cd harness
python3 -m unittest discover -s tests -v
python3 -m reg_harness.cli describe
```

### 7.2 启动图谱与检索服务

```bash
make v4-up
```

- Web 界面：http://127.0.0.1:9621  
- Neo4j：http://127.0.0.1:7474  

图为空时：`make v4-prepare`、`make v4-ingest` 等（见 Makefile）。  
使用仓内向量快照时，**向量模型与维度**须与入库时一致。

### 7.3 提问

```bash
cd harness

python3 -m reg_harness.cli --profile-env .env.gb39901_v4 \
  ask "GB 39901—2025 适用于哪两类汽车？" --max-steps 6

python3 -m reg_harness.cli --profile-env .env.gb39901_v4 \
  ask "完整列出6.11规定的五类误响应场景，并说明所有场景共同的合格判据。" \
  --max-steps 10
```

导出取证轨迹（面试/调试可用）：

```bash
python3 -m reg_harness.cli --profile-env .env.gb39901_v4 \
  ask "你的问题" --dump-trace /tmp/evidence-trail.json
```

### 7.4 跑 Benchmark（离线验证）

题目与金标在 `benchmark/data/`。典型脚本在 `benchmark/scripts/`（如跑问答结果、打分）。  
**注意：** 打分用金标；Agent 在线路径默认不应读取金标答案。报告示例：

- [pilot_6q_report.md](benchmark/results/pilot_6q_report.md)  
- [benchmark_report.md](benchmark/results/benchmark_report.md)  

### 7.5 代码调用

```python
from reg_harness import build_stack

stack = build_stack(profile_env=".env.gb39901_v4")
state = stack.ask("GB 39901 适用于哪两类汽车？", max_steps=6)
print(state.final_answer)
```

## 8. 说明

- 本项目是工程演示，不是量产知识中台，也不是认证工具。  
- 门控能降低幻觉风险，**不能保证零错误**；合规与安全结论须以正式受控文本为准。  
- Benchmark 用于 **验证与发现问题**；未冻结的总正确率勿当宣传数字。  
- 样例语料仅供学习研究，见 [NOTICE.md](NOTICE.md)。  
- 贡献方式见 [CONTRIBUTING.md](CONTRIBUTING.md)；本仓库 [MIT](LICENSE)。

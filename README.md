# EvidenceTrail

**基于知识图谱增强检索的文档取证 Agent**

---

## 1. 痛点

在 **汽车** 行业——尤其是 **智能驾驶、车载知识问答**（这里只是举例，同类需求也出现在试验规范、标定、诊断等文档里）——大家希望借助大模型回答：

- 这个工况速度阈值是多少？  
- 某种误触发怎么判定？  
- 两条试验要求差在哪里？  

但这类问题的答案，往往对应 **已经写死在文档里的确定知识**（表格数字、条款条件、合格判据），而不是开放聊天。

此时如果大模型产生 **幻觉**——捏造条款、改数字、混淆适用条件——句子仍然可能很通顺，后果却可能很严重：试验理解偏差、设计讨论跑偏，甚至触及安全相关误判。

所以核心要求是：

**必须严格按照已有的、可核对的确定知识来回答；材料不够就明确说不知道，绝不能硬编。**

## 2. 从传统 RAG，到加上 Agent

### 2.1 为什么会想到 RAG

为了让模型「有据可依」，常见做法是 **RAG（检索增强生成）**：先从知识库里找出相关文档片段，再让模型基于这些片段生成回答。这比纯靠模型背参数好。

### 2.2 传统 RAG：单次「问题 → 检索 → 回答」

传统 RAG 更像一条 **直通管道**，通常只跑 **一轮**：

```text
问题  ──►  检索（搜一批相关片段）  ──►  生成答案
              （一次定生死）              （直接出）
```

优点是简单；在汽车长文档里，一轮往往不够：

| 单次流程的短板 | 结果 |
|----------------|------|
| 只搜一次 | 跨条款、跨表的问题容易漏半截 |
| 切分不当 | 表头和数值行拆开，阈值答错 |
| 召回噪声 | 模型用「差不多相关」的段落脑补 |
| 没有「够不够」 | 证据不足也生成完整答案 → 幻觉出站 |
| 难追溯 | 用户不知道对应原文哪里 |

要「必须忠于确定知识」，既需要更好的检索组织，也需要中间有一层 **能规划、能补搜、能决定何时停手** 的控制——这就是 **Agent**。

### 2.3 本项目的关键增量：Agent（Harness 框架）

EvidenceTrail 相对传统 RAG 的首要优点，不是「多接了一个库」，而是把 **Agent 框架（代码里的 harness）** 嵌进链路：

```text
传统 RAG（单次）:
  问题 → 检索一次 → 回答

EvidenceTrail（多步 Agent）:
  问题
    →【Agent】想：还要不要搜？搜什么子问题？用图还是向量？
    → 检索 → 放进证据袋
    →【Agent】反思：够不够？不够就换问法再搜
    → 够了再基于证据袋回答
    → 代码检查：没证据 / 数字对不上原文 → 拒答
```

| | 传统 RAG | EvidenceTrail |
|--|----------|----------------|
| 流程 | **单次** 问题→检索→回答 | **多轮循环** 规划→检索→反思→再决策 |
| 谁决定搜什么 | 基本固定成整句问题 | **Agent 写子问题、选检索方式** |
| 证据不够时 | 仍常硬答 | **继续取证，或明确拒答** |
| 工程落点 | 检索 + 一次生成 | 检索底座 + **Harness 取证框架** |

> **传统 RAG = 一次检索生成；本项目 = 在 RAG 之上加了 Agent（Harness），把问答做成可规划的取证过程。**

### 2.4 知识图谱、GraphRAG 和 LightRAG

**知识图谱**把文档里的概念收成节点，把概念之间的联系收成边。  
相对纯向量库「只找语义相近的段落」，图谱更适合回答：谁验证谁、哪条要求对应哪组试验、误响应和正常制动为什么不能混成一类意思。

**GraphRAG** 可以粗理解为：检索时不只做向量相似，还利用图结构做关联扩展，再把相关原文交给大模型生成。  
常见路径是：在图上找到相关实体和关系 → 再回到原文片段 → 再生成。跨条款、跨章节的问题，往往比单次向量召回更容易串起来。

本项目构图与图+向量检索底座用的是开源 **[LightRAG](https://github.com/HKUDS/LightRAG)**。  
它负责把文档切成片段、抽实体和关系、写入图库和向量索引，并提供图增强、纯向量等查询模式。  
LightRAG 是通用底座；**领域里抽什么类型、关系允不允许、表格怎么切**，需要在应用侧自己约束。本仓库不改它的核心源码，而是用配置、定制脚本和后处理，把底座收成可用的法规知识库。

### 2.5 构图上做了哪些改进

默认 LightRAG 更偏 **通用 schema**：实体类型往往是人物、组织、地点、概念一类宽泛标签，关系也相对自由。直接用在法规、试验文档上会有明显缺点：

| 通用 schema 容易怎样 | 对问答的影响 |
|----------------------|--------------|
| 条款、试验、阈值、误响应挤在模糊类型里 | 难区分「应制动」和「不应响应」 |
| 关系端点不受约束 | 脏边多，检索沿错误边扩散 |
| 数值缺单位、条件、适用对象 | 答阈值时容易张冠李戴 |
| 表格被普通切分打断 | 表头和数据行分离，条件表题容易错 |

针对样例域，本项目把构图收成 **领域 schema + 结构切分 + 关系约束**，主要改进是：

1. **专用 schema**  
   见 [config/gb_39901_2025_schema.yml](config/gb_39901_2025_schema.yml)。  
   用条款、试验场景、阈值与条件、系统功能、功能安全链路等类型，替代通用人物/地点清单。  
   设计上强调：条款是追溯主轴；性能要求与试验方法分开再连接；正常碰撞与误响应分开；数值必须带指标、单位和适用条件。

2. **结构切分**  
   表格尽量整表或按结构单元入库，减少表被切两半。样例域上，表相关事实题可以稳定很多。

3. **关系约束**  
   抽取提示、运行时校验和后处理一起限制允许的关系类型与端点，减少类型不合法的边。  
   本地对比里，不合法关系可以从很多降到接近没有。  
   需要说明：图更干净，不等于问答自动更好，所以后面仍要靠 Agent 取证和忠于原文。

4. **检索时怎么用这张图**  
   Agent 循环里：**图上做广度搜索找路，再回源到完整原文段落答题**。  
   图命中实体或关系后，不只返回三元组摘要，还要把对应原文拉齐；作答时原文优先。默认倾向图和向量一起用，再用重排序压噪声。

验证样例是 GB 39901-2025 轻型汽车 AEBS 法规。换文档域时，主要是重写 schema 和切分策略，方法本身不绑死这一本标准。

### 2.6 设计立场

控制层只约定：Agent 要做取证、可以拒答、有步数上限、禁止无证据编造。  
不会把某道题的标准答案关键词写进规则里去刷分——那样等于瞄着靶子射箭，换一批题就失效。

题目相关的知识应在索引和原文里；Agent 根据领域说明和当前证据袋自己决定搜什么。

```text
控制层：只定角色、工具、拒答、步数
        不定某题答案关键词
              │
              ▼
        Agent 在线取证作答
              │
              │  只读
              ▼
        索引 + 原文
              │
              │  作答结束后
              ▼
        标准答案 / 参考证据 ──► 离线打分
        （提问过程中不读）
```

标准答案只用于跑完之后打分，见下一节。

## 3. 用 Benchmark 做验证

能跑通 demo 不够，还要能回答：检索有没有漏、图建得怎么样、回答是否忠于材料。  
所以项目里单独做了 **离线评测集和打分脚本**，目录在 `benchmark/`。

整条验证链路：

```text
                    ┌──────────────────┐
                    │  题目 questions  │
                    └────────┬─────────┘
                             │
           ┌─────────────────┼─────────────────┐
           ▼                 ▼                 ▼
    ┌────────────┐    ┌────────────┐    ┌────────────┐
    │ 图谱构建质量 │    │ 检索质量    │    │ 问答质量    │
    │ score_kg   │    │ score_     │    │ score_     │
    │            │    │ retrieval  │    │ answers    │
    └─────┬──────┘    └─────┬──────┘    └─────┬──────┘
          │                 │                 │
          └─────────────────┼─────────────────┘
                            ▼
                   ┌────────────────┐
                   │ 报告 / badcase  │
                   │ 再改进入库或Agent│
                   └────────────────┘
```

在线提问与离线打分要分开：

```text
【正确】先答后评

  用户问题 ──► Agent 只查索引/原文 ──► 得到回答
                                              │
                                              ▼
                         标准答案 ──► 离线打分 ──► 看对错与 badcase


【错误】开卷答题（本项目不用）

  用户问题 ──► 把标准答案/参考证据塞进提示或检索 ──► 再生成
               （等于瞄着靶子射箭，分数虚高）
```

### 3.3 标准答案怎么用

标准答案和参考证据只用于跑完之后打分、分析错在哪。  
不会在提问过程中把它们塞进检索或提示词，否则分数不能代表用户随便问时的真实表现。

在线 Agent 默认不加载这套评测数据；只有 `benchmark/scripts` 一类离线打分会用到。

### 3.4 跑下来看到的现象

在 GB 39901 样例上做过入库对比和 pilot 问答，报告在 `benchmark/results/`。大致可以画成：

```text
  图检索 ──► 证据覆盖升高
       │
       └──► 噪声也可能升高 ──► 只靠「多召回」不够
                                  │
                                  ▼
                    需要：重排序 · 原文优先 · 门控 · 多步 Agent

  结构切分 / 关系约束 ──► 图更干净、表题更稳
       │
       └──► 但不等于问答自动更好 ──► 仍要靠取证 Agent
```

用文字说就是：

- 图检索往往能提高证据覆盖，说明广搜有用。  
- 覆盖上去以后噪声也更大，答案不一定同步变好。  
- 加强关系约束后，类型不合法的关系可以明显下降。  
- 图更干净，不等于问答自动更好，多步取证和忠于原文仍然必要。  
- 表格按整表或结构切分后，表上的事实题可以稳定很多。  

完整 60 题冻结和统一总正确率还要再审题、重跑。现在仓库里主要是评测链路本身，以及 pilot、入库对比的结论，方便改完自己验。  
更细数字见 [pilot_6q_report.md](benchmark/results/pilot_6q_report.md)、[benchmark_report.md](benchmark/results/benchmark_report.md) 和 [PROJECT_STATUS.md](PROJECT_STATUS.md)。

## 4. 项目介绍：EvidenceTrail

**EvidenceTrail** 就是：Harness 取证 Agent，加上图增强检索、忠于原文的门控，以及用 Benchmark 做离线验证。

一次提问大致是：

```text
用户问题
  → Agent 规划：下一步用图还是向量、子问题怎么写
  → 检索；图命中则回源完整原文，放进证据袋
  → 重排序，把更相关的片段靠前
  → 反思：袋里够不够？不够再搜；够了就收网作答
  → 只根据证据袋生成答案，原文优先
  → 门控：没证据就拒答；关键数字必须在原文里出现
  → 输出结果和过程轨迹
```

目标：**先取证，再作答；不够就拒答。** 不是单次「搜一下就答」。

### 4.1 走题示意：6.11 五类误响应

题目：列出 6.11 规定的五类误响应场景，并说明共同合格判据。

1. Agent 用图加向量去搜 6.11 相关内容和判据；  
2. 图上命中后，回源拉完整原文，而不是只留关系短句；  
3. 袋里已经有场景和判据时，应停止空转，直接基于证据作答；  
4. 若编出来的内容和原文对不上，门控会挡住。  

过程可用 `--dump-trace` 导出，步数会随模型和索引变化。

## 5. 我做了哪些内容

### 5.1 Harness 取证框架（`harness/`）

相对「问题→检索→回答」多出来的部分主要在这里：

- 多步循环：每步选工具、写子查询，可以多轮补证  
- 只定规范、不定题解：约定要取证、可拒答，不把某题关键词写死  
- 证据袋：多轮结果累积，作答时优先用原文段落  
- Top-K 和长度限制：控制召回条数和上下文长度  
- 重排序：检索后和进袋后再排一次，压噪声  
- 图命中后补全文：不只给关系短句，把对应原文段落拉齐  
- 判断是否够用：够了就作答，减少空转  
- 门控：没证据就拒答；关键数字必须在原文里出现  
- 轨迹：步骤可导出，方便对照评测和排错  

### 5.2 文档入库与图谱（`scripts/`、`lightrag_custom/`、`config/`）

对应前面 §2.4–2.5：

- 领域 schema：`config/gb_39901_2025_schema.yml`  
- 结构切分与入库脚本：`scripts/`  
- 抽取提示与关系校验：`lightrag_custom/`  
- 通过 Docker 使用 LightRAG，不改其核心源码  

### 5.3 离线 Benchmark（`benchmark/`）

为验证整条链路而建：

- 题目与参考数据在 `benchmark/data/`  
- 打分脚本覆盖图谱、检索、问答  
- 报告在 `benchmark/results/` 下的 markdown  
- 只在事后打分用，默认不进在线提问  

### 5.4 代码与目录

```text
harness/           # Agent 主逻辑
lightrag_custom/   # 接入 LightRAG 的定制
scripts/           # 预处理、入库
benchmark/         # 评测集与打分

data/rag_storage/  # 样例最终版向量/KV 快照（约 31MB，不含图库）
compose.yaml
Makefile
```

开源的是应用层：Agent、定制和评测；LightRAG 用 Docker 拉镜像。  
不提交：含密钥的 `.env`、Neo4j 数据目录。

## 6. 环境配置

**需要：** Docker、Python 3.10+、兼容 OpenAI 协议的对话模型与向量模型 API。

```bash
git clone <本仓库>    # 建议名：evidence-trail
cd <本仓库目录>

pip install -r requirements.txt
cd harness && pip install -e . && cd ..

cp .env.example .env
# 编辑 .env：NEO4J_PASSWORD、LLM_*、EMBEDDING_*
# 切勿提交真实密钥
```

可选：配置 `RERANK_*`，检索与证据排序会更稳。  
样例工作区可用 `.env.gb39901_v4` 作为无密钥配置叠加。

## 7. 如何运行

### 7.1 离线测试（可不启大模型服务）

```bash
cd harness
python3 -m unittest discover -s tests -v
python3 -m reg_harness.cli describe
```

### 7.2 启动图谱与检索服务

```bash
make v4-up
```

- Web 界面：http://127.0.0.1:9621  
- Neo4j：http://127.0.0.1:7474  

图为空时：`make v4-prepare`、`make v4-ingest` 等（见 Makefile）。  
使用仓内向量快照时，**向量模型与维度**须与入库时一致。

### 7.3 提问

```bash
cd harness

python3 -m reg_harness.cli --profile-env .env.gb39901_v4 \
  ask "GB 39901—2025 适用于哪两类汽车？" --max-steps 6

python3 -m reg_harness.cli --profile-env .env.gb39901_v4 \
  ask "完整列出6.11规定的五类误响应场景，并说明所有场景共同的合格判据。" \
```

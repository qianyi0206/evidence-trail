# EvidenceTrail

**基于知识图谱增强检索的文档取证 Agent**

---

## 1. 痛点

在 **汽车** 行业——尤其是 **智能驾驶、车载知识问答**（这里只是举例，同类需求也出现在试验规范、标定、诊断等文档里）——大家希望借助大模型回答：

- 这个工况速度阈值是多少？  
- 某种误触发怎么判定？  
- 两条试验要求差在哪里？  

但这类问题的答案，往往对应 **已经写死在文档里的确定知识**（表格数字、条款条件、合格判据），而不是开放聊天。

此时如果大模型产生 **幻觉**——捏造条款、改数字、混淆适用条件——句子仍然可能很通顺，后果却可能很严重：试验理解偏差、设计讨论跑偏，甚至触及安全相关误判。

所以核心要求是：

**必须严格按照已有的、可核对的确定知识来回答；材料不够就明确说不知道，绝不能硬编。**

## 2. 从传统 RAG，到加上 Agent

### 2.1 为什么会想到 RAG

为了让模型「有据可依」，常见做法是 **RAG（检索增强生成）**：先从知识库里找出相关文档片段，再让模型基于这些片段生成回答。这比纯靠模型背参数好。

### 2.2 传统 RAG：单次「问题 → 检索 → 回答」

传统 RAG 更像一条 **直通管道**，通常只跑 **一轮**：

```text
问题  ──►  检索（搜一批相关片段）  ──►  生成答案
              （一次定生死）              （直接出）
```

优点是简单；在汽车长文档里，一轮往往不够：

| 单次流程的短板 | 结果 |
|----------------|------|
| 只搜一次 | 跨条款、跨表的问题容易漏半截 |
| 切分不当 | 表头和数值行拆开，阈值答错 |
| 召回噪声 | 模型用「差不多相关」的段落脑补 |
| 没有「够不够」 | 证据不足也生成完整答案 → 幻觉出站 |
| 难追溯 | 用户不知道对应原文哪里 |

要「必须忠于确定知识」，既需要更好的检索组织，也需要中间有一层 **能规划、能补搜、能决定何时停手** 的控制——这就是 **Agent**。

### 2.3 本项目的关键增量：Agent（Harness 框架）

EvidenceTrail 相对传统 RAG 的首要优点，不是「多接了一个库」，而是把 **Agent 框架（代码里的 harness）** 嵌进链路：

```text
传统 RAG（单次）:
  问题 → 检索一次 → 回答

EvidenceTrail（多步 Agent）:
  问题
    →【Agent】想：还要不要搜？搜什么子问题？用图还是向量？
    → 检索 → 放进证据袋
    →【Agent】反思：够不够？不够就换问法再搜
    → 够了再基于证据袋回答
    → 代码检查：没证据 / 数字对不上原文 → 拒答
```

| | 传统 RAG | EvidenceTrail |
|--|----------|----------------|
| 流程 | **单次** 问题→检索→回答 | **多轮循环** 规划→检索→反思→再决策 |
| 谁决定搜什么 | 基本固定成整句问题 | **Agent 写子问题、选检索方式** |
| 证据不够时 | 仍常硬答 | **继续取证，或明确拒答** |
| 工程落点 | 检索 + 一次生成 | 检索底座 + **Harness 取证框架** |

所以可以概括成：

> **传统 RAG = 一次检索生成；本项目 = 在 RAG 之上加了 Agent（Harness），把问答做成可规划的取证过程。**

图谱怎么用、原文怎么回，都是在这个 Agent 循环里服务「找得更准、答得有据」。

### 2.4 检索思想：图谱广搜，再落到原文

在 Agent 循环内部，检索侧的想法是：

1. **在知识图谱上做广度搜索**——用实体、关系扩大「该去哪找」；  
2. **再定位到对应的原文片段（chunk）**——用原文支撑事实和数字。  

**图负责找路，文负责答题。** 最终仍忠于原文，而不是只靠图上的摘要句子。

> **实现说明：** 图谱构建与图+向量检索底座使用开源 [LightRAG](https://github.com/HKUDS/LightRAG)（Docker）。  
> 本仓库重点是其上的 **Harness 取证 Agent 与工程配套**，而不是重写检索引擎。

验证样例：**GB 39901-2025 轻型汽车 AEBS 法规**。方法 **不绑定** 该标准，可换其它技术文档。

## 3. 项目介绍：EvidenceTrail

**EvidenceTrail** = **Harness 取证 Agent** + 图增强检索 + 忠于原文的门控。

一次完整提问大致是：

```text
用户问题
  →【Agent】规划：下一步用图检索还是向量检索、子问题怎么写
  →【检索】图 / 向量召回，原文进入证据袋
  →【Rerank】更相关的片段排前面
  →【Agent 反思】够不够？不够 → 继续搜；够了 → 收网作答
  →【仅基于证据袋】生成结构化答案
  →【门控】空袋拒答；关键数字必须在原文中出现
  → 输出结果 + 过程轨迹（可追溯）
```

目标始终是：**先取证，再作答；不够就拒答——而不是单次「搜一下就答」。**

## 4. 我做了哪些内容

### 4.1 Harness 取证框架（`harness/`）——相对传统 RAG 的核心增量

| 内容 | 说明 |
|------|------|
| **多步 Agent 循环** | 相对「问题→检索→回答」单次管道：每步选工具、写子查询，可多轮补证 |
| 证据袋 | 多轮结果累积；**原文 chunk 优先**，图上实体/关系主要用来导航 |
| Top-K 与长度预算 | 限制召回条数和袋容量，减少噪声占满上下文 |
| Rerank | 检索侧与袋内侧都可对片段再排序 |
| 图 → 原文 | 从图命中追溯源 chunk，补全文 |
| 反思与收网 | 判断是否空转、是否已够；够则强制作答 |
| 忠于原文门控 | 空袋拒答；答案中的关键数字须出现在证据中 |
| 轨迹 | 记录每步决策与工具结果，便于排查和评测 |

### 4.2 文档入库与图谱（`scripts/`、`lightrag_custom/`）

| 内容 | 说明 |
|------|------|
| 切分策略 | 表格尽量整表入库，减少切碎 |
| 关系约束 | 限制图谱上允许的关系类型，减少脏边 |
| 接入 LightRAG | 通过容器与定制完成建图、索引（不改 LightRAG 核心源码） |

### 4.3 离线评测（`benchmark/`）

固定题目与参考材料离线打分；**评测答案默认不进入在线提问流程**，避免「开卷」虚高。

### 4.4 代码与目录

```text
harness/           # Agent 主逻辑（Harness）
lightrag_custom/   # 接入 LightRAG 的定制
scripts/           # 预处理、入库
benchmark/         # 样例题与打分
corpus/            # 样例语料（prepared）
data/rag_storage/  # 样例最终版向量/KV 快照（约 31MB，不含图库）
compose.yaml       # Neo4j + LightRAG
Makefile           # 常用命令
```

- **不入库：** 含密钥的 `.env`、Neo4j 数据目录（约 500MB，本机 Docker 重建）  
- 示意图：[docs/architecture.svg](docs/architecture.svg)

## 5. 环境配置

**需要：** Docker、Python 3.10+、兼容 OpenAI 协议的对话模型与向量模型 API。

```bash
git clone <本仓库>    # 建议名：evidence-trail
cd <本仓库目录>

pip install -r requirements.txt
cd harness && pip install -e . && cd ..

cp .env.example .env
# 编辑 .env：NEO4J_PASSWORD、LLM_*、EMBEDDING_*
# 切勿提交真实密钥
```

可选：配置 `RERANK_*`，检索与证据排序会更稳。  
样例工作区可用 `.env.gb39901_v4` 作为无密钥配置叠加（由运行命令里的 profile 指定）。

## 6. 如何运行

### 6.1 离线测试（可不启大模型服务）

```bash
cd harness
python3 -m unittest discover -s tests -v
python3 -m reg_harness.cli describe
```

### 6.2 启动图谱与检索服务

```bash
# 仓库根目录
make v4-up
```

- Web 界面：http://127.0.0.1:9621  
- Neo4j：http://127.0.0.1:7474  

图为空时，按 Makefile 做 prepare / ingest（如 `make v4-prepare`、`make v4-ingest`）。  
使用仓内向量快照时，**向量模型与维度**须与入库时一致。

### 6.3 提问

```bash
cd harness

python3 -m reg_harness.cli --profile-env .env.gb39901_v4 \
  ask "GB 39901—2025 适用于哪两类汽车？" --max-steps 6

python3 -m reg_harness.cli --profile-env .env.gb39901_v4 \
  ask "完整列出6.11规定的五类误响应场景，并说明所有场景共同的合格判据。" \
  --max-steps 10
```

保存过程轨迹：

```bash
python3 -m reg_harness.cli --profile-env .env.gb39901_v4 \
  ask "你的问题" --dump-trace /tmp/evidence-trail.json
```

### 6.4 代码调用

```python
from reg_harness import build_stack

stack = build_stack(profile_env=".env.gb39901_v4")
state = stack.ask("GB 39901 适用于哪两类汽车？", max_steps=6)
print(state.final_answer)
```

# Harness Protocol（控制层规范）

> **定位：** 控制层定义「这层要做什么、宏观规范是什么」；模型在协议内做微观规划与内容推理；代码只 enforce 不可协商条款。  
> **反模式：** 把某篇法规、某套 pilot 题的答案线索写进路由/槽位（瞄着靶子射箭）。

本文是 harness 的 **设计北极星**。  
实现现状见 [`ARCHITECTURE.md`](ARCHITECTURE.md)；与本文不一致处标为 **协议债**，见文末对照表。

---

## 0. 一句话原则

```text
控制层 = 协议与护栏
模型   = 协议内的决策与生成
索引   = 唯一允许存放「这篇文档知识」的地方
金标   = 只用于离线打分，禁止进入在线工具链
```

换文档 / 换题目时：应只换 **workspace / 索引 / 题干**，不换控制层剧本。

---

## 1. 分层职责

| 层 | 允许做什么 | 禁止做什么 |
|----|------------|------------|
| **Protocol（本文）** | 定义目标、工具语义、证据/拒答规范、循环规则 | 写具体条款号剧本、写某题关键词清单 |
| **Code enforce** | 白名单、schema、去重、空袋、数值 grounding、max_steps | 按题型硬编码「下一工具必须是 X」 |
| **Model（决策 LLM）** | 选工具、写子 query、判断是否够答、组织答案 | 无证据编造法规事实 |
| **Tools** | 检索 / 精查 / 压缩观察 / 作答 / 结束 | 偷偷读 gold 答案或 gold evidence 列表 |
| **Index（LightRAG 等）** | 持有全文、图、向量 | — |
| **Benchmark** | 离线评分、对照实验 | 把评分逻辑泄漏进在线 Agent |

在线路径目标形态：

```text
问题
  →（可选）从题干解析出的结构信号：条款号 / 表号 / 车型 / 速度…
  → 注入 Protocol 摘要 + 工具目录 + 当前袋与观察
  → Model 输出 {thought, action, args}
  → Tool 执行 → 更新 evidence 袋
  → 重复直至 compose 过门控 或 finalize 拒答
  → 代码 gate 校验后交付
```

**不应再出现：** 大段 `if "五类" in q: slots = [6.11 铁板 行人…]` 一类贴题分支作为主路径。

## 2. 宏观目标（告诉模型：这层要做什么）

决策模型与作答模型共享同一目标陈述（可进 system prompt）：

1. **角色：** 法规取证 Agent，不是开放域百科。  
2. **目标：** 在「证据袋」可支撑的前提下给出可审计结论；否则明确拒答。  
3. **手段：** 通过工具收集带定位信息的原文证据；禁止依赖训练记忆补条款、阈值、枚举。  
4. **结束：**  
   - 证据充分 → `compose_answer`  
   - 已核验不可答 / 源中不存在 → `finalize` 且 `answerable=false`  
   - 步数耗尽仍不足 → 拒答（不可硬编）

「充分」由 **规范 + 门控** 定义，不由「是否命中某套 gold」定义。

---

## 3. 工具规范（宏观能力边界）

控制层只规定 **每个工具是什么**；**何时调用** 由模型根据题干与袋状态决定。

| action | 宏观定义 | 典型 args（契约） | 代码侧 enforce |
|--------|----------|-------------------|----------------|
| `vector_search` | 语义召回原文片段 | `query`（必填子问题）, `mode?` | 禁止空 query；可去重 |
| `graph_search` | 图增强召回（实体/关系+片段） | `query`, `mode?` | 同上 |
| `clause_lookup` | 按条款/附录编号精确取文 | `clause`（须为题干或推理得到的编号） | 无命中则观察「未找到」，不编造 |
| `table_lookup` | 按表号与条件取行/单元 | `table`, 可选 vehicle/speed/load… | 参数缺失时返回缺参说明 |
| `evidence_check` | 报告袋状态与缺口类型（非答案） | `focus?` | 只读汇总，不改金标 |
| `compose_answer` | **仅基于证据袋** 生成结构化答案 | `force?` | 空袋拒绝；过 guards |
| `finalize` | 结束：拒答或接受已校验草稿 | `answerable`, `reason`… | 写入终态 |

### 3.1 允许的「弱结构信号」（非贴题剧本）

仅当信号 **从题干解析得出** 时可作为 **软提示** 写入观察或 prompt，例如：

- 题干匹配到 `表\s*([A-Za-z]?\d+)` → 提示「题干含表号 T」  
- 题干匹配到条款形态 `\d+(\.\d+)+` → 提示「题干含条款号候选 …」  
- 题干含「是否 / 有没有 / 是否规定」→ 提示「可能需要证伪路径」  

**禁止** 作为控制层硬逻辑：

- 写死「五类误响应 → 查 6.11 + 关键词铁板/行人…」  
- 写死「80 km/h + 表 → unanswerable」而不做检索核验  
- 维护与 pilot 题一一对应的 `tools_prefer` 剧本表  

### 3.2 索引与金标隔离（硬规矩）

| 资源 | 在线 Agent | 离线 Benchmark |
|------|------------|----------------|
| LightRAG workspace | ✅ | ✅ |
| 从 **全文/结构单元** 建的定位索引 | ✅（精查应用这个） | ✅ |
| `benchmark/data` 金标 evidence / gold ids | ❌ 不得作为 lookup 唯一来源 | ✅ 仅评分 |
| 标准答案 JSON | ❌ | ✅ 仅评分 |

**协议债说明：** 当前 `EvidenceCatalog` 默认读 `benchmark/data/evidence.jsonl`，与金标同源，**违反 3.2**。后续应改为「全标准条款/表索引」或明确标记为 `eval_only` 模式。

## 4. 证据规范

### 4.1 证据袋（Evidence Bag）

- 条目至少包含：`kind`、`text`、`source_tool`；尽量含 `file_path` / 条款或表定位 / 可选 id。  
- 袋是 **唯一** 允许进入 `compose_answer` 的事实来源。  
- 压缩（compact）原则用 **形态特征**，不用专名剧本：  
  - 优先：带 `source_clause` / 来源条款、表结构、与题干 **解析出的** 数字/条款号重合的片段  
  - 降权：无定位的纯实体名罗列（可配置）  

### 4.2 缺口类型（可选；若保留 slots）

若保留槽位，只允许 **缺口类型**，不允许 **答案关键词表**：

| 类型 id（示例） | 含义（宏观） | 覆盖判定（宏观） |
|-----------------|--------------|------------------|
| `conditions` | 题干约束条件是否在证据中有对应 | 题干解析出的条件词/数字出现在袋中 |
| `threshold_or_cell` | 所问阈值/单元格 | 相关数值出现在袋中 |
| `list_items` | 枚举是否完整 | 由模型论证；门控可要求存在列表结构证据 |
| `exception_or_diff` | 例外/对比 | 存在条件分支类表述 |
| `source_anchor` | 出处条款/表 | 存在可解析定位 |
| `existence` | 条件组合是否在源中存在 | 检索后仍无对应则倾向拒答 |

**禁止：** `keywords=["6.11","铁板","行人","自行车"]` 这类把答案写进槽位。

覆盖检测优先用：题干解析 token + 证据形态；慎用固定领域词表。

## 5. 答案与拒答规范

### 5.1 答案形态（软规范 → prompt）

```json
{
  "answerable": true,
  "answer": { },
  "claims": ["原子声明…"],
  "citations": ["证据定位或袋内引用…"],
  "reason": "若 answerable=false，说明缺什么或为何判定不存在"
}
```

原则：

- 数字、阈值、速度必须能在证据原文中找到同形支撑。  
- 枚举须完整；比较须含条件/例外。  
- 题干前提在源中不成立时 → `answerable=false`，而非编造。

### 5.2 代码门控（硬 enforce）

| 规则 | 行为 |
|------|------|
| 证据袋为空且声称 answerable | 强制拒答或退回取证 |
| 答案中关键数值（如 ≥10 的量级）未出现在袋文本 | 强制拒答或 `continue_gathering` |
| action 不在白名单 | 拒绝执行，返回观察 |
| 相同工具签名短期内重复 | 拦截并提示换 query/工具 |
| 达到 max_steps 仍未合法结束 | 强制拒答，不编造 |

门控实现位置：`guards.py`、`compose_answer.py`、`loop.py`、`registry`。  
门控 **只表达协议**，不表达「第 N 题应拒答」。

## 6. 循环协议（过程规范）

```text
state = { question, evidence[], phase, step, max_steps, last_observation, trace[] }

loop:
  1. 构造 turn：Protocol 摘要 + 工具目录 + question + bag + observation
  2. model → { thought, action, args }
  3. 若 action 非法 → observation = 错误说明；continue
  4. tool.run → 更新 bag / observation / trace
  5. 若 tool 指示结束且过门控 → return final
  6. step++；若 step >= max_steps → 强制 compose(force) 或 finalize 拒答
```

### 6.1 模型每步必须遵守的输出契约

仅一个 JSON 对象：

```json
{
  "thought": "当前缺口与下一步理由（对照协议，不写训练知识当事实）",
  "action": "<白名单工具名>",
  "args": { }
}
```

### 6.2 控制层每步提供的上下文（应提供什么）

| 字段 | 目的 |
|------|------|
| 用户问题原文 | 任务 |
| 工具目录（名+描述+args schema） | 能力边界 |
| 证据袋压缩预览 | 已有事实 |
| 上一步 observation | 反馈 |
| step / max_steps | 预算 |
| （可选）题干解析出的结构信号 | 弱提示，非剧本 |

**不应再注入：** 贴题 `tools_prefer` 清单、写死的 pilot slots 关键词、gold evidence ids。

## 7. 与「规则」的关系：哪些规则合法

| 类型 | 是否合法 | 例子 |
|------|----------|------|
| 过程/安全规则 | ✅ | 空袋不作答、数值 grounding、白名单 |
| 工具语义规则 | ✅ | clause_lookup 必须带 clause |
| 题干结构解析 | ✅ | 抽出表号、条款号形态、车型 M1/N1 |
| 领域弱先验（可配置、与单题无关） | ⚠️ 慎用 | 「表」字提示可考虑 table 工具 |
| 单文档/单套题剧本 | ❌ | 五类→6.11+铁板；80km 特判 |
| 金标开卷 | ❌ | lookup 只查 gold jsonl |

合法规则的检验句：

> **换一篇从未见过的同类法规、换一套新题，这条规则是否仍成立且无需改代码？**  
> 否 → 不属于控制层，应进索引或删掉。

## 8. 评测纪律（防止瞄靶）

报告结果时须能区分：

| 配置 | 含义 | 可否当主结论 |
|------|------|--------------|
| **P0** 裸 LightRAG modes | 建图/检索基线 | 可 |
| **P1** harness + 本协议（无贴题规则、无 gold catalog） | 真协议 Agent | **主结论应用 P1** |
| **P2** harness + pilot 规则 / gold catalog | 演示上限 | 仅附录，不得宣传为泛化能力 |

当前默认实现偏 **P2 成分**；迁移目标是默认 **P1**。

## 10. 迁移顺序（不改变「实验可跑」的前提下）

1. **落盘本协议**（本文）— 统一口径。  
2. **Prompt 对齐本文 §2–§6**，删题型剧本口吻。  
3. **关掉或隔离 pilot slots/intent 硬分支**（feature flag：`HARNESS_PILOT_HEURISTICS=0` 默认关）。  
4. **Catalog 模式拆分：** `index`（全文）vs `gold`（仅 benchmark）。  
5. **跑 P1 对照分**，与旧 P2 分列报告。  
6. 再考虑删 multi-KB/Librarian 空壳或参数化 workspace（非协议核心）。

## 11. 给实现者的检查清单

提交 harness 控制层改动前自问：

- [ ] 是否引入了只对某一条款/某一题成立的分支？  
- [ ] 在线路径是否读了 gold 答案或 gold-only evidence？  
- [ ] 新增规则在「换文档」后是否仍成立？  
- [ ] 硬逻辑是否属于 §5.2 / 工具契约，而非题解？  
- [ ] Prompt 是否在讲规范，而不是在报「这题该查哪」？  

任一项为「否」→ 不允许进默认控制路径。
